## 1. 🌐 什么是URI设计


### 1.1 URI的本质理解


**📋 简单理解**
```
URI就像门牌号码：
- 现实中：北京市朝阳区xx路xx号xx楼xx室
- 网络中：https://api.example.com/users/123/orders/456

作用：让客户端能准确找到想要的资源
```

**🎯 核心概念**
- **资源(Resource)**：你想要操作的"东西"，比如用户、订单、商品
- **标识符(Identifier)**：给这个"东西"起的名字，让程序能找到它
- **统一(Uniform)**：所有资源都用同样的规则来命名

### 1.2 为什么需要设计规范


**❌ 没有规范的混乱状态**
```
混乱的API设计：
- 获取用户：/getUser?id=123
- 获取所有用户：/getAllUsers
- 创建用户：/createNewUser
- 删除用户：/deleteUserById?userId=123

问题：每个接口都不一样，开发者要记很多规则
```

**✅ 有规范的清晰状态**
```
规范的API设计：
- 获取用户：GET /users/123
- 获取所有用户：GET /users
- 创建用户：POST /users
- 删除用户：DELETE /users/123

好处：规律明显，一看就懂，容易记忆
```

## 2. 📝 资源命名核心原则


### 2.1 使用名词，不用动词


**🔸 基本原理**
```
✅ 正确做法：把URL当作"名词"
- /users        (用户这个东西)
- /orders       (订单这个东西)
- /products     (商品这个东西)

❌ 错误做法：把URL当作"动作"
- /getUsers     (获取用户这个动作)
- /createOrder  (创建订单这个动作)
- /deleteProduct (删除商品这个动作)
```

**💡 为什么要用名词**
```
HTTP方法已经表示动作了：
- GET /users     → 获取用户
- POST /users    → 创建用户  
- PUT /users/123 → 更新用户
- DELETE /users/123 → 删除用户

URL负责指向资源，HTTP方法负责表示操作
就像：动词 + 名词 = 完整的句子
```

### 2.2 集合用复数，单个用单数


**🔸 复数表示集合**
```
✅ 推荐做法：
- /users        → 用户集合（所有用户）
- /orders       → 订单集合（所有订单）
- /products     → 商品集合（所有商品）

❌ 不推荐：
- /user         → 容易让人误解
- /order        → 不够明确
- /product      → 看起来像单个商品
```

**🔸 单数表示具体资源**
```
✅ 获取具体资源：
- /users/123    → 编号为123的用户
- /orders/456   → 编号为456的订单
- /products/789 → 编号为789的商品

路径解读：
/users/123 = 用户集合中的第123号用户
```

**📊 实际应用示例**
| 操作描述 | HTTP方法 | URL路径 | 说明 |
|---------|---------|---------|------|
| 获取所有用户 | `GET` | `/users` | 复数表示集合 |
| 获取特定用户 | `GET` | `/users/123` | 单数ID表示具体项 |
| 创建新用户 | `POST` | `/users` | 向集合添加新项 |
| 更新用户 | `PUT` | `/users/123` | 更新具体项 |

### 2.3 保持命名一致性


**🔸 统一命名风格**
```
✅ 风格统一：
- /users
- /orders  
- /products
- /categories

❌ 风格混乱：
- /users
- /order-list
- /getProducts
- /category_info
```

**🔸 见名知意原则**
```
✅ 清晰明确：
- /users          → 用户
- /user-profiles  → 用户档案
- /shopping-carts → 购物车
- /order-items    → 订单项目

❌ 含糊不清：
- /data
- /info  
- /list
- /stuff
```

## 3. 🏗️ URL结构设计规范


### 3.1 大小写规范


**🔸 全小写 + 连字符**
```
✅ 推荐做法：
- /user-profiles
- /shopping-carts
- /order-history
- /product-categories

❌ 避免使用：
- /UserProfiles     (大写字母)
- /user_profiles    (下划线)
- /userProfiles     (驼峰命名)
- /USERS            (全大写)
```

**💡 为什么用小写**
```
原因说明：
1. URL大小写敏感，UserProfiles ≠ userprofiles
2. 小写更容易输入，减少出错
3. 符合Web标准，大多数网站都这样做
4. 连字符比下划线在URL中更常见
```

### 3.2 层级设计原则


**🔸 表达资源间的关系**
```
层级关系示例：
/users/123/orders          → 用户123的所有订单
/users/123/orders/456      → 用户123的订单456
/orders/456/items          → 订单456的所有商品项
/orders/456/items/789      → 订单456的商品项789

读法：
/users/123/orders = 在用户123里面找订单
```

**🔸 层级深度控制**
```
✅ 合理深度（2-3层）：
- /users/123/orders
- /categories/electronics/products
- /courses/101/lessons/5

❌ 过度嵌套（避免）：
- /companies/abc/departments/it/teams/backend/members/john/tasks/urgent
太长了，难以理解和维护
```

### 3.3 URL长度最佳实践


**📏 长度控制建议**
```
推荐长度：
- 总长度：< 2000字符（浏览器限制）
- 实际建议：< 100字符（便于阅读）
- 路径段：每段 < 50字符

示例对比：
✅ 简洁：/api/v1/users/123/orders
❌ 冗长：/api/version1/getAllOrdersForSpecificUserById/123
```

## 4. 🔧 参数设计最佳实践


### 4.1 路径参数 vs 查询参数


**🔸 路径参数：标识具体资源**
```
用法：/resources/{id}
示例：
- /users/123        → 获取用户123
- /orders/456       → 获取订单456
- /products/789     → 获取商品789

何时使用：需要指定某个具体的资源时
```

**🔸 查询参数：过滤和配置**
```
用法：/resources?key=value
示例：
- /users?age=25&city=beijing    → 筛选条件
- /products?page=2&size=10      → 分页参数
- /orders?status=pending        → 状态过滤

何时使用：需要筛选、排序、分页等操作时
```

### 4.2 常用查询参数设计


**📊 分页参数**
```
✅ 推荐方式：
- /users?page=1&size=20
- /products?offset=0&limit=10

参数说明：
- page：第几页（从1开始）
- size：每页多少条
- offset：跳过多少条（从0开始）
- limit：返回多少条
```

**🔍 搜索和过滤**
```
✅ 常见搜索参数：
- /users?search=张三
- /products?category=electronics
- /orders?status=completed&date=2024-01-01

✅ 排序参数：
- /users?sort=created_at
- /products?sort=price&order=desc
```

**⚠️ 查询参数注意事项**
```
💡 小贴士：
- 参数名要清晰明确
- 使用标准的分页、排序参数名
- 特殊字符要进行URL编码
- 可选参数要有默认值
```

## 5. ❌ 常见设计错误避坑


### 5.1 动词错误用法


**❌ 常见错误示例**
```
错误的URL设计：
- /getUsers          → 应该用 GET /users
- /createUser        → 应该用 POST /users  
- /updateUser/123    → 应该用 PUT /users/123
- /deleteUser/123    → 应该用 DELETE /users/123
- /searchUsers       → 应该用 GET /users?search=xxx
```

**✅ 正确的改进方案**
```
REST标准做法：
- GET /users         → 获取用户列表
- POST /users        → 创建新用户
- GET /users/123     → 获取指定用户
- PUT /users/123     → 更新指定用户
- DELETE /users/123  → 删除指定用户
```

### 5.2 命名不一致错误


**❌ 混乱的命名风格**
```
风格不统一的问题：
- /users             (复数)
- /product           (单数)
- /order-list        (带后缀)
- /getCategories     (带动词)
- /user_profiles     (下划线)

开发者要记住各种不同的规则，容易出错
```

**✅ 统一的命名规范**
```
保持一致的风格：
- /users             (复数名词)
- /products          (复数名词)
- /orders            (复数名词)  
- /categories        (复数名词)
- /user-profiles     (复数名词 + 连字符)

所有资源都遵循相同规则，容易记忆
```

### 5.3 过度设计错误


**❌ 冗长复杂的URL**
```
过度设计的例子：
/api/v1/company/departments/engineering/teams/backend/employees/john/current-projects/active/tasks/high-priority

问题：
- 太长难记
- 层级过深  
- 难以维护
- 性能较差
```

**✅ 简化的设计方案**
```
简化方案：
- /employees/john/tasks?priority=high&status=active
- /teams/backend/tasks?assignee=john&priority=high

好处：
- 简洁明了
- 容易理解
- 便于维护
- 性能更好
```

## 6. 📋 核心要点总结


### 6.1 设计核心原则


```
🔸 名词不用动词：URL指向资源，不是动作
🔸 复数表示集合：/users 表示用户集合
🔸 保持一致性：所有接口遵循相同规则
🔸 见名知意：URL要能直观理解含义
🔸 简洁明了：避免过长的URL
🔸 全小写：使用小写字母和连字符
```

### 6.2 实用设计模板


**📋 基础CRUD模板**
```
资源集合操作：
- GET /resources          → 获取资源列表
- POST /resources         → 创建新资源

单个资源操作：
- GET /resources/{id}     → 获取指定资源
- PUT /resources/{id}     → 更新指定资源  
- DELETE /resources/{id}  → 删除指定资源
```

**📋 嵌套资源模板**
```
父子关系资源：
- GET /users/{id}/orders     → 获取用户的订单
- POST /users/{id}/orders    → 为用户创建订单
- GET /orders/{id}/items     → 获取订单的商品项
```

### 6.3 记忆口诀


```
📝 设计原则口诀：
名词复数做资源，动词交给HTTP方法
层级清晰不过深，参数分类要明确
大小写要保持一致，见名知意最重要
```

### 6.4 常用检查清单


**✅ URL设计检查项**
- [ ] 使用名词而非动词
- [ ] 集合资源使用复数形式
- [ ] 保持全小写 + 连字符
- [ ] URL长度合理（< 100字符）
- [ ] 层级深度适中（2-3层）
- [ ] 命名风格统一一致
- [ ] 见名知意，含义清晰

**🎯 实际应用建议**
- **新手阶段**：严格遵循基础规范，养成良好习惯
- **进阶阶段**：根据业务特点灵活调整，但保持团队一致性
- **团队协作**：制定统一的API设计规范文档，定期评审

**核心记忆**：
- URL是资源的门牌号，要清晰准确
- 名词 + HTTP方法 = 完整的API操作
- 一致性比完美性更重要
- 简单易懂胜过复杂精巧
