## 1. 🌍 数据保护法规概述


### 1.1 合规加密的重要性


**什么是合规加密**：指按照法律法规和行业标准要求，对敏感数据进行保护的加密措施。

```
数据泄露的代价：

罚款损失：
• GDPR最高可罚营收4%或2000万欧元
• 国内网络安全法最高可罚100万元

业务影响：
• 客户信任度下降
• 品牌声誉受损
• 业务中断风险
• 法律诉讼成本
```

### 1.2 全球主要数据保护法规


| 法规名称 | **适用范围** | **核心要求** | **违规处罚** |
|---------|------------|-------------|-------------|
| 🇪🇺 **GDPR** | `欧盟及处理欧盟个人数据` | `数据保护设计、加密传输` | `营收4%或2000万欧元` |
| 🇺🇸 **CCPA** | `加州居民个人信息` | `数据透明度、安全措施` | `每次违规最高7500美元` |
| 🇨🇳 **网络安全法** | `中国境内网络运营者` | `数据本地化、安全保护` | `最高100万元罚款` |
| 🇨🇳 **个保法** | `处理个人信息活动` | `最小化原则、安全保障` | `最高5000万元罚款` |

### 1.3 合规加密基本原则


```
设计安全原则（Privacy by Design）：
┌─────────────────┐
│   业务需求       │
```

## 2. 🛡️ GDPR数据保护要求


### 2.1 GDPR核心概念


**个人数据定义**：任何能够直接或间接识别自然人的信息。

```sql
-- GDPR保护的个人数据示例
CREATE TABLE user_profile (
    user_id INT PRIMARY KEY,
    email VARCHAR(255),           -- 直接标识符
    phone VARCHAR(20),           -- 直接标识符  
    ip_address VARCHAR(45),      -- 间接标识符
    device_fingerprint TEXT,     -- 间接标识符
    location_data JSON,          -- 敏感个人数据
    health_info TEXT             -- 特殊类别个人数据
);
```

### 2.2 GDPR加密要求


**第32条 - 处理安全性**：要求采取适当的技术措施，包括数据加密。

```
GDPR加密要求框架：

数据传输加密：
• 使用TLS 1.2+协议
• 端到端加密通信
• 安全密钥交换

数据存储加密：
• 静态数据加密（AES-256）
• 数据库透明加密
• 备份文件加密

访问控制：
• 基于角色的访问控制
• 多因素身份认证
• 最小权限原则
```

### 2.3 GDPR合规实施


```sql
-- GDPR合规的MySQL配置示例
-- 1. 启用数据加密
SET GLOBAL innodb_encryption_threads = 4;
SET GLOBAL default_table_encryption = ON;

-- 2. 创建加密表
CREATE TABLE gdpr_user_data (
    id INT AUTO_INCREMENT PRIMARY KEY,
    user_id VARCHAR(36) NOT NULL,
    encrypted_email VARBINARY(512),    -- 加密存储
    encrypted_phone VARBINARY(256),    -- 加密存储
    consent_status ENUM('granted', 'withdrawn') NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
) ENGINE=InnoDB ENCRYPTION='Y';

-- 3. 数据处理日志表（审计要求）
CREATE TABLE data_processing_log (
    log_id BIGINT AUTO_INCREMENT PRIMARY KEY,
    user_id VARCHAR(36),
    action_type ENUM('create', 'read', 'update', 'delete'),
    data_category VARCHAR(100),
    legal_basis VARCHAR(100),
    processor_id VARCHAR(100),
    timestamp TIMESTAMP DEFAULT CURRENT_TIMESTAMP
) ENGINE=InnoDB ENCRYPTION='Y';
```

> 💡 **GDPR关键提示**：必须能够证明合规性，而不仅仅是声称合规。这要求完整的文档记录和技术实施证据。

## 3. 💳 PCI DSS支付卡标准


### 3.1 PCI DSS概述


**PCI DSS**：支付卡行业数据安全标准，保护持卡人数据安全的技术和操作要求。

```
PCI DSS 12项要求：

构建和维护安全网络：
要求1: 安装和维护防火墙配置
要求2: 不使用供应商提供的默认密码

保护持卡人数据：
要求3: 保护存储的持卡人数据  ← 核心加密要求
要求4: 加密传输持卡人数据    ← 核心加密要求

维护漏洞管理程序：
要求5: 保护所有系统免受恶意软件侵害
要求6: 开发和维护安全系统和应用程序

...（其他要求）
```

### 3.2 持卡人数据加密要求


**要求3详解**：存储的持卡人数据必须使用强加密算法保护。

```sql
-- PCI DSS合规的支付数据存储
CREATE TABLE payment_cards (
    card_token VARCHAR(32) PRIMARY KEY,     -- 令牌化
    encrypted_pan VARBINARY(256),           -- 加密主账号（只存前4后4位明文）
    pan_first4 CHAR(4),                     -- 前4位（用于显示）
    pan_last4 CHAR(4),                      -- 后4位（用于显示）
    encrypted_expiry VARBINARY(128),        -- 加密有效期
    cardholder_name_hash VARCHAR(64),       -- 持卡人姓名哈希
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
) ENGINE=InnoDB ENCRYPTION='Y';

-- 敏感认证数据表（禁止存储）
-- ❌ 不能存储的数据：
-- • 完整磁道数据
-- • CVV/CVC验证码
-- • PIN验证数据
```

### 3.3 PCI DSS密钥管理


```
密钥管理要求：

密钥生成：

• 密钥长度符合行业标准（AES-256）
• 定期更换密钥

密钥存储：
• 密钥分割存储

密钥分发：
• 安全的密钥交换协议
```

## 6. 🏷️ 数据分类分级策略


### 6.1 数据分类原则


**数据分类**：根据数据的敏感程度和业务重要性进行分类管理。

```
数据分类体系：

公开数据（Public）：
• 可以公开发布的信息
• 无保密要求
• 如：产品介绍、公司新闻

内部数据（Internal）：
• 仅限内部使用的信息
• 基本保密要求
• 如：员工通讯录、内部通知

机密数据（Confidential）：
• 高度敏感的商业信息
• 严格保密要求
• 如：财务报表、客户资料

绝密数据（Top Secret）：
• 极度敏感的核心信息
• 最高保密要求
• 如：商业机密、个人隐私
```

### 6.2 数据分级实施


```sql
-- 数据分级标签系统
CREATE TABLE data_classification_labels (
    label_id INT AUTO_INCREMENT PRIMARY KEY,
    classification_level ENUM('public', 'internal', 'confidential', 'top_secret'),
    encryption_required BOOLEAN DEFAULT FALSE,
    retention_period_days INT,
    access_control_policy JSON,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 带分级标签的业务表
CREATE TABLE business_documents (
    doc_id VARCHAR(36) PRIMARY KEY,
    title VARCHAR(200),
    encrypted_content LONGBLOB,             -- 根据分级决定是否加密
    classification_level ENUM('public', 'internal', 'confidential', 'top_secret'),
    
    -- 自动分级字段
    contains_pii BOOLEAN DEFAULT FALSE,     -- 是否包含个人信息
    contains_financial BOOLEAN DEFAULT FALSE, -- 是否包含财务信息
    contains_trade_secret BOOLEAN DEFAULT FALSE, -- 是否包含商业机密
    
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    
    INDEX idx_classification (classification_level),
    INDEX idx_pii (contains_pii)
) ENGINE=InnoDB;

-- 分级触发器示例
DELIMITER //
CREATE TRIGGER auto_classify_data 
BEFORE INSERT ON business_documents
FOR EACH ROW
BEGIN
    -- 自动确定加密要求
    IF NEW.classification_level IN ('confidential', 'top_secret') THEN
        -- 这里应该调用加密函数处理content
        SET NEW.encrypted_content = AES_ENCRYPT(NEW.encrypted_content, 'encryption_key');
    END IF;
END//
DELIMITER ;
```

## 7. 🔍 加密审计机制


### 7.1 审计日志设计


**审计目标**：记录所有涉及敏感数据的操作，确保可追溯性。

```sql
-- 综合审计日志表
CREATE TABLE comprehensive_audit_log (
    audit_id BIGINT AUTO_INCREMENT PRIMARY KEY,
    
    -- 基本信息
    timestamp TIMESTAMP(6) DEFAULT CURRENT_TIMESTAMP(6),
    session_id VARCHAR(64),
    user_id VARCHAR(100),
    user_role VARCHAR(50),
    
    -- 操作信息
    operation_type ENUM('SELECT', 'INSERT', 'UPDATE', 'DELETE', 'ADMIN'),
    database_name VARCHAR(64),
    table_name VARCHAR(64),
    affected_rows INT,
    
    -- 数据分类信息
    data_classification ENUM('public', 'internal', 'confidential', 'top_secret'),
    contains_encrypted_data BOOLEAN DEFAULT FALSE,

    access_granted BOOLEAN,

    -- 网络信息

    INDEX idx_timestamp (timestamp),
    INDEX idx_user_operation (user_id, operation_type),
    INDEX idx_compliance (compliance_framework),
    INDEX idx_classification (data_classification)
) ENGINE=InnoDB ENCRYPTION='Y';
```

### 7.2 实时监控告警


```sql
-- 异常行为监控视图
CREATE VIEW suspicious_activities AS
SELECT 
    user_id,
    COUNT(*) as operation_count,
    COUNT(DISTINCT table_name) as tables_accessed,
    SUM(CASE WHEN data_classification = 'top_secret' THEN 1 ELSE 0 END) as secret_access,
    MAX(timestamp) as last_activity
FROM comprehensive_audit_log 
WHERE timestamp >= DATE_SUB(NOW(), INTERVAL 1 HOUR)
GROUP BY user_id
HAVING operation_count > 1000 
    OR tables_accessed > 50 
    OR secret_access > 10;

-- 合规违规检查
CREATE VIEW compliance_violations AS
SELECT 
    audit_id,
    user_id,
    operation_type,
    table_name,
    timestamp,
    'Unauthorized access to encrypted data' as violation_type
FROM comprehensive_audit_log
WHERE contains_encrypted_data = TRUE 
    AND access_granted = FALSE
    AND timestamp >= DATE_SUB(NOW(), INTERVAL 24 HOUR);
```

### 7.3 审计报告生成


```sql
-- 月度合规报告存储过程
DELIMITER //
CREATE PROCEDURE generate_monthly_compliance_report(
    IN report_month DATE,
    IN compliance_type VARCHAR(20)
)
BEGIN
    DECLARE report_start_date DATE;
    DECLARE report_end_date DATE;
    
    SET report_start_date = DATE_FORMAT(report_month, '%Y-%m-01');
    SET report_end_date = LAST_DAY(report_month);
    
    -- 创建临时报告表
    CREATE TEMPORARY TABLE temp_compliance_report AS
    SELECT 
        DATE(timestamp) as activity_date,
        compliance_framework,
        operation_type,
        data_classification,
        COUNT(*) as operation_count,
        COUNT(DISTINCT user_id) as unique_users,
        SUM(CASE WHEN access_granted = FALSE THEN 1 ELSE 0 END) as failed_access_attempts
    FROM comprehensive_audit_log
    WHERE timestamp BETWEEN report_start_date AND report_end_date
        AND FIND_IN_SET(compliance_type, compliance_framework) > 0
    GROUP BY DATE(timestamp), compliance_framework, operation_type, data_classification;
    
    -- 返回报告结果
    SELECT * FROM temp_compliance_report ORDER BY activity_date, operation_count DESC;
    
    DROP TEMPORARY TABLE temp_compliance_report;
END//
DELIMITER ;
```

## 8. 🏗️ 企业级加密实施


### 8.1 加密技术选型


**技术选型原则**：安全性、性能、兼容性、可管理性的平衡。

```
加密算法选择指南：

对称加密：
✅ AES-256-GCM（推荐）
• 性能优秀，安全性高
• 支持认证加密
• 广泛标准支持

✅ ChaCha20-Poly1305
• 适合移动设备
• 抗时序攻击
• 软件实现性能好

❌ DES/3DES（已废弃）
• 密钥长度不足
• 存在安全漏洞

非对称加密：
✅ RSA-4096
• 成熟稳定
• 广泛支持

✅ ECC P-384
• 密钥长度短
• 性能优秀

哈希算法：
✅ SHA-256/SHA-3
❌ MD5/SHA-1（已废弃）
```

### 8.2 密钥管理架构


```
企业密钥管理层次：

层级1: 主密钥（Master Key）
┌─────────────────────────┐
│      HSM/KMS 存储        │ ← 硬件安全模块保护
│    Root Encryption Key   │
└─────────────────────────┘
            │
            ▼
层级2: 数据加密密钥（DEK）
┌─────────────────────────┐
│    Database Keys        │ ← 用于数据库加密
│    Application Keys     │ ← 用于应用层加密  
│    Backup Keys         │ ← 用于备份加密
└─────────────────────────┘
            │
            ▼
层级3: 业务密钥
┌─────────────────────────┐
│    Table-level Keys     │ ← 表级加密
│    Column-level Keys    │ ← 列级加密
│    Row-level Keys      │ ← 行级加密
└─────────────────────────┘
```

### 8.3 加密实施路径


```sql
-- 阶段性加密实施计划

-- 第一阶段：基础设施加密
-- 1. 启用传输加密
SET GLOBAL ssl_cert = '/path/to/server-cert.pem';
SET GLOBAL ssl_key = '/path/to/server-key.pem';
SET GLOBAL ssl_ca = '/path/to/ca-cert.pem';

-- 2. 启用静态加密
```
