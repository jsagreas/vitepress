## 1. 🎯 项目架构搭建概述


### 1.1 什么是项目架构搭建


**通俗理解**：就像盖房子前要先画图纸、打地基一样，做项目前也要先搭好"骨架"

```
盖房子的步骤：              做项目的步骤：
1. 设计图纸                1. 设计项目结构
2. 打地基                  2. 搭建基础框架
3. 搭建框架                3. 配置开发环境
4. 装修细节                4. 编写业务代码

核心理念：先把"房子框架"搭好，后面装修就容易了
```

### 1.2 架构搭建的核心目标


**🔸 清晰的结构**
```
好的项目架构就像整理好的衣柜：
- 每类衣服放在固定位置（代码分层清晰）
- 需要什么一眼就能找到（易于维护）
- 新买的衣服知道往哪放（易于扩展）
```

**🔸 规范的管理**
- **依赖管理**：统一管理项目需要的工具包（就像统一采购装修材料）
- **配置管理**：把各种配置文件放在合适位置（像把工具分类摆放）
- **版本管理**：记录每次修改（像装修日志，出问题能回溯）

### 1.3 SSM项目的典型架构


```
SSM项目的三层结构：
┌─────────────────────────┐
│   表现层（Controller）   │  ← 接收用户请求，返回结果
├─────────────────────────┤
│   业务层（Service）      │  ← 处理业务逻辑
├─────────────────────────┤
│   持久层（Mapper/DAO）   │  ← 操作数据库
└─────────────────────────┘

类比：就像餐厅的三个部门
- 服务员（Controller）：接待客人，传递菜单
- 厨师（Service）：根据菜单做菜
- 仓库管理员（Mapper）：管理食材（数据）
```

## 2. 🏗️ Maven项目创建


### 2.1 Maven是什么


**通俗理解**：Maven就像是项目的"管家"

```
没有Maven的时候：
- 手动下载jar包 → 麻烦
- 不同版本冲突 → 头疼
- 团队协作困难 → 抓狂

有了Maven之后：
- 自动下载依赖 → 省心
- 统一版本管理 → 安心
- 一键构建项目 → 开心
```

### 2.2 创建Maven项目的方式


**🔸 方式一：IDEA快速创建（推荐新手）**

```
操作步骤：
1. File → New → Project
2. 选择Maven → 勾选"Create from archetype"
3. 选择 maven-archetype-webapp（Web项目模板）
4. 填写项目信息：
   - GroupId：com.example（公司域名倒写）
   - ArtifactId：ssm-project（项目名称）
   - Version：1.0-SNAPSHOT（版本号）
```

**🔸 方式二：Maven命令创建**

```bash
# 使用Maven原型创建Web项目
mvn archetype:generate \
  -DgroupId=com.example \
  -DartifactId=ssm-project \
  -DarchetypeArtifactId=maven-archetype-webapp \
  -DinteractiveMode=false
```

### 2.3 项目坐标理解


**Maven坐标的三要素**

| 要素 | 含义 | 示例 | 通俗理解 |
|------|------|------|----------|
| **GroupId** | 组织/公司标识 | `com.example` | 就像你的"姓氏" |
| **ArtifactId** | 项目名称 | `ssm-project` | 就像你的"名字" |
| **Version** | 版本号 | `1.0-SNAPSHOT` | 就像你的"年龄" |

```
完整坐标示例：
com.example:ssm-project:1.0-SNAPSHOT

理解：example公司的ssm-project项目的1.0快照版本
类比：张三家的老二的18岁照片
```

## 3. 📁 目录结构设计


### 3.1 标准SSM项目目录结构


```
ssm-project/
├── src/
│   ├── main/
│   │   ├── java/                    # Java源代码目录
│   │   │   └── com/example/
│   │   │       ├── controller/      # 控制层（接待员）
│   │   │       ├── service/         # 业务层（厨师）
│   │   │       │   └── impl/        # 业务实现类
│   │   │       ├── mapper/          # 持久层（仓库管理）
│   │   │       ├── entity/          # 实体类（数据模型）
│   │   │       ├── dto/             # 数据传输对象
│   │   │       ├── vo/              # 视图对象
│   │   │       └── config/          # 配置类
│   │   ├── resources/               # 资源文件目录
│   │   │   ├── mapper/              # MyBatis映射文件
│   │   │   ├── application.yml      # 主配置文件
│   │   │   ├── spring/              # Spring配置
│   │   │   ├── mybatis-config.xml   # MyBatis配置
│   │   │   └── logback.xml          # 日志配置
│   │   └── webapp/                  # Web资源目录
│   │       ├── WEB-INF/
│   │       │   └── web.xml          # Web配置文件
│   │       ├── static/              # 静态资源
│   │       │   ├── css/
│   │       │   ├── js/
│   │       │   └── images/
│   │       └── pages/               # 页面文件
│   └── test/                        # 测试代码目录
│       ├── java/
│       └── resources/
├── target/                          # 编译输出目录
├── pom.xml                          # Maven配置文件
└── README.md                        # 项目说明文档
```

### 3.2 目录结构设计原则


**🔰 分层清晰原则**

```
为什么要分层？
就像超市分区：
- 蔬菜区（controller）：顾客接触的地方
- 加工区（service）：处理商品的地方
- 仓库区（mapper）：存储货物的地方

好处：
✓ 职责明确：每层只做自己的事
✓ 易于维护：改一层不影响其他层
✓ 便于测试：可以单独测试每一层
```

**🔰 包命名规范**

| 包名 | 用途 | 命名规则 | 示例 |
|------|------|----------|------|
| **controller** | 控制器 | 复数形式 | `UserController` |
| **service** | 业务接口 | 复数形式 | `UserService` |
| **service.impl** | 业务实现 | Impl结尾 | `UserServiceImpl` |
| **mapper** | 数据访问 | Mapper结尾 | `UserMapper` |
| **entity** | 实体类 | 单数形式 | `User` |
| **dto** | 数据传输对象 | DTO结尾 | `UserDTO` |

### 3.3 资源文件组织


**配置文件分类管理**

```
resources/
├── application.yml          # 主配置（统一入口）
├── application-dev.yml      # 开发环境配置
├── application-prod.yml     # 生产环境配置
├── mapper/                  # SQL映射文件
│   └── UserMapper.xml
├── static/                  # 静态资源
│   ├── css/
│   ├── js/
│   └── images/
└── templates/               # 页面模板
    └── index.html

理解：
- 主配置是"总开关"
- 环境配置是"分开关"
- 需要哪个环境就激活哪个配置
```

## 4. 📦 依赖管理配置


### 4.1 pom.xml核心配置


**什么是pom.xml**

```
pom.xml就像项目的"采购清单"：
- 列出项目需要的所有工具包
- Maven会自动帮你下载
- 还会自动处理依赖关系

类比：
你要做菜（开发项目）
pom.xml就是菜谱上的食材清单
Maven就是帮你去超市采购的助手
```

**基础配置示例**

```xml
<?xml version="1.0" encoding="UTF-8"?>
<project xmlns="http://maven.apache.org/POM/4.0.0"
         xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
         xsi:schemaLocation="http://maven.apache.org/POM/4.0.0 
         http://maven.apache.org/xsd/maven-4.0.0.xsd">
    
    <!-- 基础信息 -->
    <modelVersion>4.0.0</modelVersion>
    <groupId>com.example</groupId>
    <artifactId>ssm-project</artifactId>
    <version>1.0-SNAPSHOT</version>
    <packaging>war</packaging>
    
    <!-- 属性配置（统一管理版本号） -->
    <properties>
        <spring.version>5.3.20</spring.version>
        <mybatis.version>3.5.10</mybatis.version>
        <mysql.version>8.0.29</mysql.version>
        <java.version>1.8</java.version>
    </properties>
</project>
```

### 4.2 SSM核心依赖


**依赖分类管理**

```xml
<dependencies>
    <!-- Spring核心依赖（框架基础） -->
    <dependency>
        <groupId>org.springframework</groupId>
        <artifactId>spring-webmvc</artifactId>
        <version>${spring.version}</version>
    </dependency>
    
    <!-- MyBatis依赖（数据库操作） -->
    <dependency>
        <groupId>org.mybatis</groupId>
        <artifactId>mybatis</artifactId>
        <version>${mybatis.version}</version>
    </dependency>
    
    <!-- MySQL驱动（连接数据库） -->
    <dependency>
        <groupId>mysql</groupId>
        <artifactId>mysql-connector-java</artifactId>
        <version>${mysql.version}</version>
    </dependency>
</dependencies>
```

**🔸 依赖理解**

```
依赖关系类比：
做蛋糕需要：
- 面粉（Spring框架）         → 基础框架
- 鸡蛋（MyBatis框架）        → 数据库工具
- 烤箱（MySQL驱动）          → 数据库连接
- 糖（其他工具包）           → 辅助功能

Maven会：
1. 自动下载这些"食材"
2. 检查"食材"是否新鲜（版本兼容）
3. 把"食材"放到正确位置（classpath）
```

### 4.3 版本管理最佳实践


**🎯 使用properties统一管理版本**

```xml
<!-- 为什么要统一管理版本？ -->
<properties>
    <!-- 集中定义版本号 -->
    <spring.version>5.3.20</spring.version>
</properties>

<!-- 使用版本号 -->
<dependency>
    <groupId>org.springframework</groupId>
    <artifactId>spring-core</artifactId>
    <version>${spring.version}</version>  <!-- 引用统一版本 -->
</dependency>

好处：
✓ 修改一处，全局生效（省事）
✓ 避免版本冲突（安全）
✓ 一眼看出所有版本（清晰）
```

## 5. 🏢 多模块项目结构


### 5.1 为什么需要多模块


**单模块vs多模块对比**

```
单模块项目（小餐馆）：
ssm-project
└── 所有代码都在一起
    └── 优点：简单直接
    └── 缺点：代码耦合，难以复用

多模块项目（连锁餐厅）：
ssm-parent（总部）
├── ssm-common（共用设施）
├── ssm-service（厨房）
└── ssm-web（门店）
    └── 优点：模块独立，易于管理
    └── 缺点：结构复杂

选择建议：
• 小项目（功能少）→ 单模块
• 大项目（功能多）→ 多模块
• 企业项目 → 多模块
```

### 5.2 多模块项目结构设计


**标准多模块结构**

```
ssm-parent/                      # 父工程（管理者）
├── pom.xml                      # 父pom（统一管理依赖版本）
├── ssm-common/                  # 公共模块（工具箱）
│   ├── pom.xml
│   └── src/
│       └── main/java/
│           └── com/example/common/
│               ├── utils/       # 工具类
│               ├── exception/   # 异常类
│               └── constant/    # 常量类
├── ssm-entity/                  # 实体模块（数据模型）
│   ├── pom.xml
│   └── src/
│       └── main/java/
│           └── com/example/entity/
│               └── User.java
├── ssm-dao/                     # 持久层模块（数据访问）
│   ├── pom.xml
│   └── src/
│       ├── main/java/
│       │   └── com/example/mapper/
│       └── main/resources/mapper/
├── ssm-service/                 # 业务层模块（业务逻辑）
│   ├── pom.xml
│   └── src/
│       └── main/java/
│           └── com/example/service/
└── ssm-web/                     # 表现层模块（控制器）
    ├── pom.xml
    └── src/
        └── main/
            ├── java/
            │   └── com/example/controller/
            └── webapp/
```

### 5.3 父子模块依赖关系


**模块依赖图**

```
依赖方向（从上到下）：
          ssm-parent（父工程）
                ↓
    ┌───────────┼───────────┐
    ↓           ↓           ↓
ssm-common  ssm-entity  ssm-dao
                ↓           ↓
            ssm-service ←───┘
                ↓
             ssm-web

理解：
- 箭头方向表示依赖方向
- 下层模块依赖上层模块
- 上层模块不知道下层的存在

类比：
- parent像公司总部（制定规范）
- common像后勤部（提供工具）
- entity像设计部（定义模型）
- dao像仓库（管理数据）
- service像生产部（业务处理）
- web像销售部（对外服务）
```

**父pom配置示例**

```xml
<project>
    <!-- 声明为父工程 -->
    <groupId>com.example</groupId>
    <artifactId>ssm-parent</artifactId>
    <version>1.0-SNAPSHOT</version>
    <packaging>pom</packaging>
    
    <!-- 声明子模块 -->
    <modules>
        <module>ssm-common</module>
        <module>ssm-entity</module>
        <module>ssm-dao</module>
        <module>ssm-service</module>
        <module>ssm-web</module>
    </modules>
    
    <!-- 统一版本管理 -->
    <properties>
        <spring.version>5.3.20</spring.version>
    </properties>
</project>
```

## 6. ⚙️ 配置文件组织


### 6.1 配置文件分类


**配置文件的作用**

```
配置文件就像"说明书"：
- 告诉程序怎么连接数据库
- 告诉程序监听哪个端口
- 告诉程序使用什么日志级别

类比：
- 家电说明书：教你怎么使用
- 配置文件：教程序怎么运行
```

**核心配置文件清单**

| 配置文件 | 作用 | 格式 | 位置 |
|----------|------|------|------|
| **application.yml** | 主配置文件 | YAML | resources/ |
| **spring-mvc.xml** | SpringMVC配置 | XML | resources/spring/ |
| **spring-service.xml** | Spring业务层配置 | XML | resources/spring/ |
| **mybatis-config.xml** | MyBatis核心配置 | XML | resources/ |
| **UserMapper.xml** | SQL映射文件 | XML | resources/mapper/ |
| **logback.xml** | 日志配置 | XML | resources/ |

### 6.2 application.yml主配置


**配置文件示例**

```yaml
# 服务器配置
server:
  port: 8080                    # 监听端口（就像门牌号）
  servlet:
    context-path: /ssm          # 访问路径（就像楼层号）

# 数据源配置（数据库连接信息）
spring:
  datasource:
    driver-class-name: com.mysql.cj.jdbc.Driver
    url: jdbc:mysql://localhost:3306/ssm_db
    username: root              # 数据库用户名
    password: 123456            # 数据库密码

# MyBatis配置
mybatis:
  mapper-locations: classpath:mapper/*.xml    # SQL文件位置
  type-aliases-package: com.example.entity    # 实体类包路径
  configuration:
    map-underscore-to-camel-case: true        # 下划线转驼峰

# 日志配置
logging:
  level:
    com.example.mapper: debug                 # SQL日志级别
```

**配置项理解**

```
配置项含义解释：

🔸 server.port = 8080
   → 程序监听8080端口
   → 访问地址：http://localhost:8080

🔸 spring.datasource.url
   → 数据库地址：localhost（本机）
   → 端口：3306（MySQL默认端口）
   → 数据库名：ssm_db

🔸 mybatis.mapper-locations
   → 告诉MyBatis去哪找SQL文件
   → classpath表示resources目录
   → *.xml表示所有xml文件

🔸 logging.level
   → 设置日志详细程度
   → debug：打印详细信息（开发时用）
   → info：只打印重要信息（生产时用）
```

### 6.3 环境配置分离


**多环境配置管理**

```
不同环境需要不同配置：
开发环境（dev）：
- 数据库：localhost
- 日志：详细（debug）
- 端口：8080

测试环境（test）：
- 数据库：test.example.com
- 日志：一般（info）
- 端口：8081

生产环境（prod）：
- 数据库：prod.example.com
- 日志：简要（warn）
- 端口：80
```

**配置文件组织**

```yaml
# application.yml（主配置）
spring:
  profiles:
    active: dev           # 激活哪个环境

---
# application-dev.yml（开发环境）
spring:
  datasource:
    url: jdbc:mysql://localhost:3306/ssm_dev

---
# application-prod.yml（生产环境）
spring:
  datasource:
    url: jdbc:mysql://prod.example.com:3306/ssm_prod
```

## 7. 🚀 启动类与包结构


### 7.1 Spring Boot启动类


**启动类的作用**

```
启动类就像"点火器"：
- 启动整个Spring容器
- 扫描所有组件
- 初始化配置

类比：
启动汽车需要：
1. 插入钥匙（启动类）
2. 点火（@SpringBootApplication）
3. 发动引擎（Spring容器启动）
```

**启动类示例**

```java
package com.example;

import org.springframework.boot.SpringApplication;
import org.springframework.boot.autoconfigure.SpringBootApplication;

/**
 * SSM项目启动类
 * @SpringBootApplication = 三个注解的组合
 */
@SpringBootApplication
public class SsmApplication {
    
    public static void main(String[] args) {
        // 启动Spring容器
        SpringApplication.run(SsmApplication.class, args);
        System.out.println("SSM项目启动成功！");
    }
}
```

**注解理解**

```
@SpringBootApplication 包含三个注解：

1. @SpringBootConfiguration
   → 标记这是配置类（相当于配置文件）

2. @EnableAutoConfiguration  
   → 自动配置（Spring自动帮你配置好）

3. @ComponentScan
   → 扫描组件（找到所有Controller、Service等）

原理：
就像开餐厅：
- @Configuration：制定经营方案
- @EnableAutoConfiguration：自动配备设施
- @ComponentScan：招聘所有员工
```

### 7.2 包结构规划


**标准包结构**

```java

├── SsmApplication.java             # 启动类（放在根包）
├── controller                      # 控制层
│   ├── UserController.java

├── service                         # 业务层
│   ├── UserService.java
│   └── impl
│       └── UserServiceImpl.java
├── mapper                          # 持久层
│   └── UserMapper.java
├── entity                          # 实体类
│   └── User.java
├── dto                             # 数据传输对象
│   └── UserDTO.java
├── vo                              # 视图对象
│   └── UserVO.java
```

**包命名规范**

```
包命名三原则：
1. 全小写（lowercase）
2. 见名知意（一看就懂）
3. 分层清晰（职责明确）

好的命名：
✓ com.example.controller     （控制层）
✓ com.example.service.impl   （业务实现）
✓ com.example.mapper         （数据访问）

不好的命名：
✗ com.example.ctrl           （缩写不明确）
✗ com.example.business       （太笼统）
✗ com.example.db             （不规范）
```

### 7.3 组件扫描配置


**自动扫描原理**

```
Spring怎么找到组件？

默认扫描规则：
1. 从启动类所在包开始
2. 扫描该包及所有子包
3. 找到带注解的类（@Controller、@Service等）

示例：
启动类：com.example.SsmApplication
会扫描：com.example.*（所有子包）

不会扫描：
- com.other.*（其他包）
- 没有注解的类
```

**自定义扫描**

```java
// 指定扫描包
@SpringBootApplication
@ComponentScan(basePackages = {
    "com.example.controller",
    "com.example.service",
    "com.other.utils"           // 扫描其他包
})
public class SsmApplication {
    // ...
}
```

### 8.2 Git版本管理

```
Git就像"时光机"：

- 可以回到任何历史版本
- 团队协作不会冲突

- 草稿2（第二次提交）
- 草稿3（第三次提交）
- 需要时可以回到任何版本
```

**基本操作流程**

```bash

# 3. 提交到本地仓库

# 4. 查看状态
git status                   # 查看修改状态
```
