## 1. 🚨 应急预案基础概念


### 1.1 什么是应急预案


**🔸 核心定义**
```
应急预案：当系统出现故障时，快速恢复服务的标准化操作手册
目标：最短时间内恢复业务，减少损失
本质：把经验变成流程，把流程变成习惯
```

**💡 为什么需要应急预案**
想象你家突然停电了：
- ❌ **没预案**：慌乱找手电筒，不知道断路器在哪，浪费大量时间
- ✅ **有预案**：5秒内点亮应急灯，1分钟内恢复供电

系统故障也是一样 - **时间就是金钱，预案就是救命稻草**

### 1.2 应急预案的核心要素


**📋 预案必备组成部分**
```
🎯 故障识别：如何快速判断问题类型
⚡ 响应流程：谁来做、怎么做、做什么
🔧 操作步骤：具体的命令和操作
📞 联系方式：关键人员电话和通讯录
⏰ 时间要求：每个步骤的时限要求
🔄 回滚方案：万一操作失败怎么办
```

### 1.3 应急响应等级划分


| 等级 | **严重程度** | **影响范围** | **响应时间** | **典型场景** |
|------|------------|-------------|-------------|-------------|
| 🔴 **P0级** | `核心业务中断` | `全部用户` | `5分钟内` | `主数据库宕机` |
| 🟡 **P1级** | `重要功能异常` | `部分用户` | `30分钟内` | `支付系统故障` |
| 🟢 **P2级** | `次要功能问题` | `少量用户` | `2小时内` | `日志系统异常` |

> 💡 **记忆技巧**：P0像急诊科，P1像门诊，P2像体检中心

## 2. 🖥️ 服务器宕机应急处理


### 2.1 服务器宕机的常见原因


**🔍 故障原因分析**
```
硬件故障：
• 电源故障 → 服务器突然关机
• 内存损坏 → 系统蓝屏或重启
• 硬盘故障 → 无法启动系统
• CPU过热 → 自动关机保护

软件故障：
• 内核崩溃 → 系统死机
• 服务异常 → 进程占用资源过多
• 配置错误 → 系统无法启动
• 磁盘空间满 → 服务无法写入文件
```

**🎯 快速诊断方法**
```bash
# 第一时间检查项目
ping 服务器IP地址        # 网络是否通
ssh 用户名@服务器IP      # 是否能登录
systemctl status        # 系统服务状态
df -h                   # 磁盘空间使用情况
free -m                 # 内存使用情况
```

### 2.2 服务器宕机应急处理流程


**⚡ 标准处理步骤**

**第一阶段：快速评估（5分钟内）**
```
🔸 Step 1: 确认故障范围
   - 只是一台服务器还是多台？
   - 影响了哪些业务服务？
   - 用户访问是否受影响？

🔸 Step 2: 尝试远程连接
   - SSH是否能登录？
   - IPMI/iDRAC管理口是否正常？
   - 控制台是否有响应？

🔸 Step 3: 启动临时替代方案
   - 切换到备用服务器
   - 启用负载均衡故障转移
   - 通知相关业务方
```

**第二阶段：故障恢复（30分钟内）**

**🔧 硬件故障处理**
```
电源故障：
1. 检查电源线连接
2. 更换电源线或插座
3. 联系机房检查供电

无法启动：
1. 通过IPMI查看硬件状态
2. 检查内存、硬盘指示灯
3. 尝试安全模式启动
4. 必要时更换硬件
```

**💻 软件故障处理**
```bash
# 系统无响应时的处理
sudo reboot                    # 尝试重启
sudo systemctl --failed       # 查看失败的服务
sudo journalctl -xe           # 查看系统日志
sudo systemctl rescue         # 进入救援模式
```

### 2.3 预防服务器宕机的措施


**📊 监控指标设置**
```
CPU使用率：超过80%告警，90%紧急
内存使用率：超过85%告警，95%紧急
磁盘使用率：超过80%告警，90%紧急
系统负载：超过CPU核数的2倍告警
网络连通性：ping失败3次告警
```

> ⚠️ **注意事项**：监控阈值不是越严格越好，太敏感会产生大量误报

## 3. 🌐 网络中断故障恢复


### 3.1 网络故障的分类和识别


**🔍 网络故障类型**
```
物理层故障：
• 网线松动或损坏 → 网卡指示灯不亮
• 交换机端口故障 → 单台设备无网络
• 光纤断裂 → 整个网段中断

网络层故障：
• 路由配置错误 → 能ping通网关，不能访问外网
• DNS解析异常 → 能ping IP，不能访问域名
• 防火墙规则过严 → 部分服务无法访问

应用层故障：
• 服务端口未监听 → 连接被拒绝
• 负载过高 → 连接超时
• SSL证书过期 → HTTPS访问失败
```

### 3.2 网络故障诊断工具


**🛠️ 常用诊断命令**
```bash
# 网络连通性测试
ping -c 4 baidu.com           # 测试外网连通性
ping -c 4 192.168.1.1        # 测试网关连通性
ping -c 4 127.0.0.1          # 测试本地回环

# 路由追踪
traceroute baidu.com          # 查看数据包路径
mtr baidu.com                 # 连续监测网络质量

# 端口和服务检测
telnet 服务器IP 端口          # 测试端口连通性
nmap -p 22,80,443 服务器IP    # 扫描端口开放情况
netstat -tulnp               # 查看本机监听端口
```

### 3.3 网络中断应急恢复流程


**⚡ 网络故障处理步骤**

```
阶段一：快速定位（10分钟内）
┌─────────────────────────────┐
│ 1. 确认故障范围             │
│    • 单台还是多台设备？     │
│    • 内网还是外网问题？     │
│                             │
│ 2. 检查物理连接             │
│    • 网线是否插好？         │
│    • 指示灯是否正常？       │
│                             │
│ 3. 测试网络各层             │
│    • ping本地回环           │
│    • ping网关               │
│    • ping外网               │
└─────────────────────────────┘
```

**🔧 分层修复策略**
```
物理层修复：
• 重新插拔网线
• 更换网线或交换机端口
• 检查交换机电源和状态

网络层修复：
• 重启网络服务：systemctl restart network
• 检查IP配置：ip addr show
• 重新获取IP：dhclient -r && dhclient

应用层修复：
• 重启相关服务：systemctl restart nginx
• 检查防火墙：iptables -L
• 验证配置文件语法
```

### 3.4 网络故障预防措施


**📡 网络冗余设计**
```
双线路接入：
主线路 + 备用线路，自动切换

双网卡绑定：
bond0 = eth0 + eth1，提高可靠性

多路径路由：
配置多个网关，负载均衡
```

> 💡 **实用建议**：网络问题80%都是物理连接问题，先检查网线再看配置

## 4. 💾 存储系统故障预案


### 4.1 存储故障的常见类型


**🔍 存储系统故障分析**
```
硬盘故障：
• 单盘故障 → RAID降级运行
• 多盘故障 → 数据可能丢失
• 控制器故障 → 整个阵列不可用

文件系统故障：
• 文件系统损坏 → 无法挂载
• 磁盘空间耗尽 → 服务无法写入
• 权限问题 → 应用无法访问

存储网络故障：
• SAN网络中断 → 存储不可访问  
• 光纤通道故障 → 性能下降
• 交换机故障 → 存储路径中断
```

### 4.2 存储故障诊断方法


**🛠️ 存储健康检查**
```bash
# 查看磁盘状态
lsblk                         # 查看磁盘分区情况
df -h                         # 查看磁盘使用率
mount | grep -v tmpfs         # 查看挂载情况

# 硬盘健康检测
smartctl -a /dev/sda         # 查看硬盘SMART信息
badblocks -v /dev/sda        # 检测坏块

# RAID状态检查
cat /proc/mdstat             # 软RAID状态
megacli -AdpAllInfo -aALL    # 硬RAID状态（需要工具）
```

### 4.3 存储故障应急处理流程


**⚡ 存储故障处理优先级**

| 故障类型 | **紧急程度** | **处理策略** | **预期恢复时间** |
|---------|-------------|-------------|----------------|
| 🔴 **多盘故障** | `极高` | `立即启用备份系统` | `30分钟内` |
| 🟡 **单盘故障** | `高` | `更换故障盘，重建RAID` | `2小时内` |
| 🟢 **磁盘空间满** | `中` | `清理日志，扩容存储` | `1小时内` |

**🔧 具体处理步骤**

**单盘故障处理：**
```bash
# 1. 确认故障盘
cat /proc/mdstat
# md0 : active raid5 sdb1[1] sdc1[2] sdd1[3]
# 如果看到 [F] 标记，说明该盘故障

# 2. 标记故障盘并移除
mdadm --manage /dev/md0 --fail /dev/sdb1
mdadm --manage /dev/md0 --remove /dev/sdb1

# 3. 物理更换硬盘后重建
mdadm --manage /dev/md0 --add /dev/sdb1
# 系统会自动开始重建，可以通过 cat /proc/mdstat 查看进度
```

**磁盘空间满处理：**
```bash
# 快速清理大文件
find /var/log -name "*.log" -size +100M -exec rm {} \;
find /tmp -atime +7 -exec rm -rf {} \;
journalctl --vacuum-time=7d    # 清理7天前的系统日志

# 临时扩容（如果有空间）
lvextend -L +10G /dev/mapper/vg0-root
resize2fs /dev/mapper/vg0-root
```

### 4.4 存储系统备份策略


**📂 多层备份方案**
```
本地备份：每天增量，每周全量
异地备份：每周同步到远程机房
云备份：重要数据定期上传云存储
```

> ⚠️ **重要提醒**：RAID不是备份！RAID只能防止硬件故障，不能防止误删除和病毒

## 5. 🗄️ 数据库异常处理流程


### 5.1 数据库故障的常见场景


**🔍 数据库故障分类**
```
性能类故障：
• 查询慢，响应超时 → 索引缺失或锁等待
• 连接数耗尽 → 应用连接池配置问题
• 内存不足 → 缓存命中率低

可用性故障：
• 服务无法启动 → 配置错误或数据文件损坏
• 主从同步中断 → 网络问题或从库延迟
• 数据库崩溃 → 硬件故障或软件bug

数据类故障：
• 数据丢失 → 误删除或硬件故障
• 数据不一致 → 主从数据不同步
• 表空间满 → 磁盘空间不足
```

### 5.2 数据库故障快速诊断


**🛠️ MySQL故障诊断步骤**
```bash
# 1. 检查数据库服务状态
systemctl status mysql
ps -ef | grep mysql

# 2. 查看数据库连接情况
mysql -e "SHOW PROCESSLIST;"
mysql -e "SHOW STATUS LIKE 'Threads_connected';"

# 3. 检查错误日志
tail -f /var/log/mysql/error.log
tail -f /var/log/mysql/slow.log

# 4. 查看数据库性能指标
mysql -e "SHOW ENGINE INNODB STATUS\G"
mysql -e "SHOW VARIABLES LIKE 'innodb_buffer_pool_size';"
```

### 5.3 数据库异常应急处理


**⚡ 按严重程度分级处理**

**🔴 P0级：数据库完全不可用**
```
应急响应流程：
第1分钟：启动只读从库，保证基础查询
第5分钟：分析主库故障原因
第15分钟：修复主库或切换到从库
第30分钟：恢复所有服务正常
```

**🟡 P1级：性能严重下降**
```bash
# 快速处理慢查询
mysql -e "SHOW PROCESSLIST;" | grep "Query" | awk '{print $1}' | xargs -I {} mysql -e "KILL {};"

# 临时增加连接数
mysql -e "SET GLOBAL max_connections = 500;"

# 重启数据库服务（谨慎操作）
systemctl restart mysql
```

**🟢 P2级：部分功能异常**
```sql
-- 检查表空间使用情况
SELECT 
    table_schema,
    SUM(data_length + index_length) / 1024 / 1024 AS 'DB Size MB'
FROM information_schema.tables
GROUP BY table_schema;

-- 优化表和重建索引
OPTIMIZE TABLE 表名;
ANALYZE TABLE 表名;
```

### 5.4 数据库主从切换预案


**🔄 主从切换操作流程**

```
准备阶段检查：
┌────────────────────────────┐
│ 1. 检查从库同步状态        │
│    SHOW SLAVE STATUS\G     │
│                            │
│ 2. 确认数据一致性          │
│    比较主从数据差异        │
│                            │
│ 3. 通知应用停止写操作      │
│    避免切换期间数据丢失    │
└────────────────────────────┘

切换执行步骤：
1. 停止应用写入
2. 等待从库同步完成
3. 提升从库为主库
4. 修改应用配置指向新主库
5. 启动应用服务
```

> 💡 **切换建议**：主从切换要在业务低峰期进行，提前通知用户

## 6. ⚙️ 应用服务故障恢复


### 6.1 应用服务故障的典型表现


**🔍 应用故障现象**
```
服务无响应：
• HTTP请求超时 → 可能是进程死锁
• 连接被拒绝 → 服务端口未监听
• 502/503错误 → 后端服务不可用

性能问题：
• 响应时间变长 → CPU或内存不足
• 部分功能异常 → 依赖服务故障
• 间歇性错误 → 负载过高导致

资源问题：
• 内存泄漏 → 进程内存持续增长
• CPU占用过高 → 死循环或算法问题
• 磁盘IO过高 → 大量日志写入
```

### 6.2 应用服务诊断工具


**🛠️ 服务状态检查命令**
```bash
# 服务运行状态
systemctl status 服务名
ps aux | grep 服务名
pgrep -f 服务名

# 端口监听情况  
netstat -tulnp | grep :8080
ss -tulnp | grep :8080

# 服务日志查看
journalctl -u 服务名 -f
tail -f /var/log/应用名/application.log

# 资源使用情况
top -p $(pgrep 服务名)
htop -p $(pgrep 服务名)
```

### 6.3 应用服务故障处理流程


**⚡ 标准恢复步骤**

**第一步：快速恢复服务（5分钟内）**
```bash
# 尝试重启服务
systemctl restart 服务名

# 如果重启失败，查看日志
journalctl -u 服务名 --lines=50

# 检查配置文件语法
nginx -t                    # Nginx配置检查
httpd -t                   # Apache配置检查
java -jar app.jar --check  # Java应用配置检查
```

**第二步：问题根因分析（30分钟内）**
```bash
# 检查系统资源
free -m                    # 内存使用情况
df -h                      # 磁盘空间
iostat -x 1 5             # IO统计
vmstat 1 5                # 系统整体状态

# 分析应用日志
grep -i error /var/log/app/app.log | tail -20
grep -i exception /var/log/app/app.log | tail -20
```

### 6.4 不同类型应用的处理方法


**💻 Web服务器故障处理**
```bash
# Nginx故障处理
nginx -s reload            # 重新加载配置
nginx -s reopen           # 重新打开日志文件
systemctl restart nginx   # 重启服务

# Apache故障处理  
httpd -k graceful         # 优雅重启
apachectl configtest      # 配置测试
systemctl restart httpd   # 重启服务
```

**☕ Java应用故障处理**
```bash
# 查看Java进程
jps -v                     # 显示Java进程

# 分析内存使用
jmap -histo PID           # 内存中对象统计
jstack PID                # 线程堆栈信息

# GC分析
jstat -gc PID 1000        # 每秒显示GC情况
```

> ⚠️ **注意事项**：重启是万能药，但要先保留现场信息用于后续分析

## 7. ⚖️ 负载均衡器故障切换


### 7.1 负载均衡器的作用和重要性


**🔍 负载均衡器的核心功能**
```
流量分发：将用户请求分散到多台服务器
健康检查：自动剔除故障服务器
故障转移：单点故障时自动切换
SSL卸载：统一处理HTTPS加密解密
```

**💡 简单理解**
负载均衡器就像餐厅的服务员总管：
- 客人来了，安排到空闲的餐桌（流量分发）
- 哪桌服务有问题，就不安排新客人（健康检查）  
- 某个服务员请假，其他人顶上（故障转移）

### 7.2 负载均衡器故障的识别


**🔍 常见故障现象**
```
用户侧表现：
• 网站访问慢或无法访问
• 部分功能正常，部分功能异常
• 间歇性服务中断

技术侧现象：
• 健康检查失败告警
• 某些后端服务器无流量
• 负载均衡器日志出现错误
```

### 7.3 Nginx负载均衡故障处理


**🔧 Nginx负载均衡配置检查**
```bash
# 检查配置文件
nginx -t

# 查看upstream状态（需要status模块）
curl http://localhost/nginx_status

# 重新加载配置
nginx -s reload
```

**⚡ 故障切换步骤**
```
手动故障切换流程：
1. 登录负载均衡器
2. 编辑upstream配置
3. 注释故障服务器或标记为down
4. 重新加载配置
5. 验证流量分发正常

配置示例：
upstream backend {
    server 192.168.1.10:8080 weight=1;
    server 192.168.1.11:8080 weight=1;  
    server 192.168.1.12:8080 down;  # 故障服务器
}
```

### 7.4 高可用负载均衡方案


**🔄 双机热备方案**
```
架构设计：
主负载均衡器 + 备用负载均衡器
使用Keepalived实现VIP漂移

故障切换过程：
1. 主负载均衡器故障
2. Keepalived检测到故障
3. VIP自动漂移到备机
4. 用户访问无感知切换
```

> 💡 **最佳实践**：负载均衡器本身也要做高可用，避免单点故障

## 8. 📊 监控系统失效处理


### 8.1 监控系统的重要性


**🔍 监控系统的作用**
```
故障预警：在问题发生前提前告警
性能分析：帮助优化系统性能  
故障定位：快速找到问题根源
趋势分析：预测系统发展趋势
```

**💡 形象比喻**
监控系统就像人体的神经系统：
- 感知异常（告警）
- 传达信息（通知）
- 帮助诊断（数据分析）
- 指导治疗（故障处理）

### 8.2 监控系统故障的影响


**⚠️ 监控失效的严重后果**
```
盲飞状态：
• 不知道系统是否正常
• 用户投诉才发现故障  
• 无法评估故障影响范围
• 故障恢复时间延长

运维困难：
• 缺乏性能数据支持
• 无法进行容量规划
• 故障原因难以定位
• 服务质量无法保证
```

### 8.3 监控系统故障处理


**🛠️ 常见监控系统组件**
```bash
# Zabbix监控系统
systemctl status zabbix-server
systemctl status zabbix-agent

# Prometheus监控系统  
systemctl status prometheus
systemctl status grafana-server

# 系统自带监控工具

systemctl status snmpd      # SNMP监控
```

**⚡ 监控系统恢复步骤**
```
第一步：恢复核心监控（10分钟内）
1. 重启监控服务进程
2. 检查配置文件完整性  
3. 确认数据采集正常
4. 验证告警功能可用

第二步：数据完整性检查（30分钟内）  
1. 检查历史数据是否丢失
2. 重新同步监控配置
3. 补充缺失的监控项
4. 测试告警通知渠道
```

### 8.4 监控系统备用方案

## 9. 💾 备份系统启用流程


### 9.1 备份系统的分类和特点


**🔍 备份系统类型**
```
按备份方式：
• 完整备份：备份所有数据，恢复快但占空间大
• 增量备份：只备份变化数据，节省空间但恢复慢  
• 差异备份：备份与上次完整备份的差异

按存储位置：
• 本地备份：恢复快，但不防火灾等意外
• 异地备份：安全性高，但恢复慢
• 云备份：成本低，扩展性好
```

### 9.2 备份系统健康检查


**🛠️ 备份状态验证**
```bash
# 检查备份任务状态
crontab -l | grep backup     # 查看定时备份任务
systemctl status backup      # 查看备份服务状态

# 验证备份文件完整性
ls -lh /backup/              # 查看备份文件大小和时间
md5sum /backup/latest.tar.gz # 检查文件完整性
tar -tf /backup/latest.tar.gz | wc -l  # 统计备份文件数量

# 测试备份可用性  
tar -tzf /backup/latest.tar.gz > /dev/null  # 测试压缩包是否损坏
```

### 9.3 备份系统启用应急流程


**⚡ 备份恢复标准流程**

**🔴 紧急数据恢复（生产环境）**
```
恢复前准备：
1. 评估数据丢失范围
2. 选择最合适的备份点
3. 准备恢复环境
4. 通知相关人员

恢复执行步骤：
1. 停止相关应用服务
2. 从备份中恢复数据
3. 验证数据完整性
4. 启动应用服务
5. 业务功能测试
```

**🟡 测试环境恢复（验证备份）**
```bash
# 1. 创建临时恢复目录
mkdir -p /tmp/restore_test
cd /tmp/restore_test

# 2. 解压备份文件
tar -xzf /backup/database_backup_20240915.tar.gz

# 3. 验证关键文件
ls -la restored_data/
file restored_data/important_file.db

# 4. 数据库恢复测试（MySQL示例）
mysql -u root -p test_db < restored_data/database.sql
mysql -u root -p -e "SELECT COUNT(*) FROM test_db.users;"
```

### 9.4 备份策略优化建议


**📊 3-2-1备份规则**
```
3个副本：原始数据 + 2个备份副本
2种介质：本地硬盘 + 磁带或云存储  
1个异地：至少有1个备份在异地

时间规划：
• 每天增量备份
• 每周完整备份  
• 每月异地备份
• 每季度备份验证
```

**⚙️ 自动化备份脚本示例**
```bash
#!/bin/bash
DATE=$(date +%Y%m%d_%H%M%S)
BACKUP_DIR="/backup"
SOURCE_DIR="/data"

# 创建备份
tar -czf ${BACKUP_DIR}/backup_${DATE}.tar.gz ${SOURCE_DIR}

# 验证备份
if [ $? -eq 0 ]; then
    echo "备份成功: backup_${DATE}.tar.gz"
    # 删除7天前的备份
    find ${BACKUP_DIR} -name "backup_*.tar.gz" -mtime +7 -delete
else
    echo "备份失败!" | mail -s "备份告警" admin@company.com
fi
```

> 💡 **重要提醒**：备份不测试等于没备份，要定期验证备份的可用性

## 10. 📋 核心要点总结


### 10.1 应急预案制定的关键原则


**🎯 核心理念**
```
🔸 时间就是金钱：每分钟的延迟都有成本
🔸 预案胜过临时抱佛脚：有备无患
🔸 标准化操作：减少人为错误
🔸 分级响应：不同故障不同处理方式
🔸 持续改进：每次故障都是学习机会
```

### 10.2 故障处理的通用步骤


**⚡ 万能故障处理流程**
```
第1步：快速评估影响范围（5分钟内）
    ↓
第2步：启动临时解决方案（减少损失）
    ↓  
第3步：根因分析和彻底修复
    ↓
第4步：验证修复效果
    ↓
第5步：总结经验和改进预案
```

### 10.3 各类故障的处理要点


| 故障类型 | **关键处理要点** | **预防措施** |
|---------|-----------------|-------------|
| 🖥️ **服务器宕机** | `快速切换到备用服务器` | `双机热备，资源监控` |
| 🌐 **网络中断** | `分层诊断，物理连接优先检查` | `链路冗余，多路径设计` |
| 💾 **存储故障** | `数据安全第一，立即启用备份` | `RAID配置，定期备份` |
| 🗄️ **数据库异常** | `保护数据完整性，主从切换` | `读写分离，性能调优` |
| ⚙️ **应用故障** | `先重启恢复服务，后分析原因` | `健康检查，资源限制` |
| ⚖️ **负载均衡故障** | `手动切换后端服务器` | `双机热备，健康检查` |
| 📊 **监控失效** | `恢复监控服务，补充缺失数据` | `监控的监控，备用方案` |
| 💾 **备份问题** | `验证备份完整性，选择恢复点` | `3-2-1策略，定期测试` |

### 10.4 应急预案演练的重要性


**🎭 为什么要演练**
```
发现问题：演练中发现预案的不足
熟悉流程：让团队熟悉操作步骤  
提升技能：提高故障处理能力
建立信心：减少真实故障时的恐慌
```

**📅 演练频率建议**
```
月度演练：核心业务故障场景
季度演练：复杂的综合性故障
年度演练：全公司大规模演练
```

### 10.5 实用的应急响应检查清单


**📝 故障响应检查清单**
```
□ 立即评估影响范围和严重程度
□ 按预案执行相应等级的响应流程  
□ 及时通知相关人员和业务方
□ 启动临时措施减少业务损失
□ 详细记录故障现象和处理过程
□ 根因分析，制定彻底解决方案
□ 验证修复效果，确认问题解决
□ 故障复盘，更新和优化预案
```
